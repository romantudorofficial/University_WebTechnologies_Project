3. External Interface Requirements

3.1 User Interfaces

The MaMa - Manners Matter website will interact with two main user groups: customers and administrators. Customers are people who want to learn more about netiquette, and administrators are responsible for adding new lessons to the website. Users will be required to create an account and log in to access the website's features.

The website's user interface will be designed to follow industry-standard web design principles and best practices. The layout will be intuitive, with easy-to-use navigation and clear calls-to-action. The site's design will be consistent across all pages, with a header that includes the site's logo and a sign-out button that returns the user to the login page.

The user interface will be web-based, and will be designed to be responsive and accessible across a range of devices and screen sizes. The site will include standard buttons and functions, such as a help button and navigation buttons, as well as keyboard shortcuts and other accessibility features as needed.

The user interface will be designed to meet accessibility standards and ensure that all users can access the site's content and features, regardless of their abilities. In addition, the website will comply with relevant data privacy regulations to ensure that user data is protected.

3.2 Hardware Interfaces

The MaMa - Manners Matter website does not have any specific hardware requirements and can run on any device that has a web browser and an internet connection. The software does not interact with any specific hardware components and is not dependent on any particular hardware communication protocols.

3.3 Software Interfaces

The MaMa - Manners Matter website will interact with a database where information about the users will be stored, including their email address, password, name, other specific information about them, and the number of points they have accumulated. This database will be used for the log in component, for creating the leaderboard, and for keeping track of which lessons a user has already completed.

The website is built using HTML, PHP, and CSS programming languages, and will interact with the database using SQL queries. The website will not interact with any other specific software components or libraries.

To access the database, the website will connect to it using the appropriate credentials and communication protocols. The data items that will be exchanged between the website and the database include user login information, user profile information, lesson completion data, and leaderboard data. The purpose of each data item is to facilitate user authentication, personalize the user experience, and provide feedback on user progress.

3.4 Communications Interfaces

The MaMa - Manners Matter website will require communications functions to enable user interactions such as email notifications and feedback submissions. The following communication protocols and standards will be used:

  • Email: The website will send email notifications to users to confirm their account creation and password reset requests. The email protocol used will be SMTP.
  • Web Browser: The website will interact with web browsers to deliver its content to users. The supported web browsers will be Chrome, Firefox, Safari, and Edge. The website will follow web standards such as HTML5 and CSS3 to ensure compatibility across different web browsers.
  • HTTP: The website will use HTTP protocol for communication between the web server and client web browsers.
  • HTTPS: The website will use HTTPS protocol to encrypt and secure data transmission between the web server and client web browsers.

In addition, the website will comply with data privacy and security regulations such as GDPR and CCPA. Any data collected from users will be stored securely and will not be shared with third-party services without user consent. Any communication security or encryption issues will be addressed using industry-standard encryption protocols.

4. System Features

4.1 User Authentication

4.1.1 Description and Priority

Users can create an account and log in to access their account information, progress, and to complete lessons. This feature is of high priority.

4.1.2 Stimulus/Response Sequences

  • User enters login credentials and clicks the Log In button.
  • System verifies user credentials.
  • If credentials are valid, system displays the main page.
  • If credentials are invalid, system displays error message and prompts user to enter valid credentials.

4.1.3 Functional Requirements

  • The system shall provide a user registration form.
  • The system shall store user login credentials in a secure database.
  • The system shall verify user credentials before granting access to user account.
  • The system shall allow users to reset their password in case they forget it.
  • The system shall display appropriate error messages if invalid credentials are entered.

4.2 Lesson Browsing

4.2.1 Description and Priority

Users will be able to browse and view available lessons. This feature is of high priority.

4.2.2 Stimulus/Response Sequences

  • User clicks on "Go To Lessons" button in the main page.
  • System displays a list of available lesson categories in the header.
  • User clicks on a specific lesson category.
  • System displays a list of available lessons in the respective category.
  • User clicks on a specific lesson.
  • System displays lesson content.

4.2.3 Functional Requirements

  • The system shall display a list of available lessons.
  • The system shall display the lesson content in a readable and organized format.
  • The system shall allow users to mark a lesson as completed.
  • The system shall update the user's progress and points accordingly.

4.3 Leaderboard

4.3.1 Description and Priority

The leaderboard is a feature that displays the top 10 users based on the number of points they have accumulated through completing lessons. It is of medium priority as it is not critical to the core functionality of the system but provides a gamification aspect to the learning experience.

4.3.2 Stimulus/Response Sequences

  • User clicks on "Scoreboard" button on the main page.
  • System displays the leaderboard page with the top 10 users listed in descending order based on points accumulated, as well as the user's own score and placement if they are not placed in top 10.

4.3.3 Functional Requirements

  • The system shall keep track of the number of points accumulated by each user.
  • The system shall calculate the total number of points for each user based on the number of lessons completed and their difficulty level.
  • The system shall display the top 10 users with their names and number of points accumulated.
  • The system shall update the leaderboard in real-time as users complete lessons and accumulate points.
  • The system shall provide a way for users to view their own rank on the leaderboard.

4.4 Admin Lesson Management

4.4.1 Description and Priority

Admin Lesson Management refers to the ability of the admin to add, edit, and delete lessons on the site. This feature is of high priority.

4.4.2 Stimulus/Response Sequences

  • System prompts the admin to enter their login credentials.
  • Admin logs in to the system.
  • Admin clicks on "Go To Lessons" button in the main page.
  • System display the "Admin Commands" option in the header.
  • Admin clicks on the aforementioned option.
  • System displays the management page.

4.4.3 Functional Requirements

  • The system shall provide a secure login for admins to access the lesson management functionality.
  • The system shall allow admins to add new lessons to the site, including a title, description, and content.
  • The system shall allow admins to delete lessons from the site, with a confirmation prompt to prevent accidental deletion.
  • The system shall allow admins to edit existing lessons, including the ability to modify the title, description, and content.
  • The system shall maintain the integrity of the database when adding, editing, or deleting lessons.
  • The system shall display a success message to the admin after a lesson has been successfully added, edited, or deleted.
  • The system shall ensure that only authenticated admins have access to the lesson management functionality.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

The project is implemented in PHP which is a superior option for integrating or customizing any popular CMS for our project. Even if it is slower than Node.js , PHP comes with more advantages as it has been stated. We will use a database as a storage for many parameters: from user data to questions answers. It is very important for us to have a dynamic web application, that is why, we will use them in order to offer the high wanted performance. For suggestions we will use the property of PHP to send the user directly to the specific content that he would like to have an idea on how to behave in a specific context.

5.2 Safety Requirements

The only ways that could be with loss of data might be when deleting a category of lessons or a specific lesson. But everything is handled by the back-end, so there should not be any issues. In case of something, the other admins could redo everything.

5.3 Security Requirements

First of all, the application will use verification of the roles: if it is an admin or if it has an account. This is for being sure that no person will use it without having an account, so the other users will be protected.

Secondly, the passwords will be encrypted, this will provide safety for each user that their account will NOT be stolen by another person. It is very important to keep them safe.

Eventually, if the application grows, it will be changed to HTTPS for more security than the current state that we are in.

5.4 Software Quality Attributes

It is able to get new categories of lessons and to add lessons to every existent category. It is also able to delete categories and lessons. This should show that it is a flexible application down for changes of its content.

But the application, might have additional future features, such as being able to modify the navigation bar. That would be a nice attribute for implementation.

It could be adapted really quickly if we would use a new page for doing all this hard-work automatically.

6. Other Requirements

6.1 Database Requirements

The database would be a relational one. This would help us implement faster joins between tables in order to find the content faster. The application will be divided in more tables depending on types: user, info about questions, lessons, categories. Then we will just create connections between them through ids.

6.2 Internationalization Requirements

The application is already written in English, but over time we could implement a way of displaying the content in different languages through the help of Google. At first, might be used only important langauges [ spoken by a higher number of people].

6.3 Legal Requirements

We will keep hidden the users' data. It is important for us that they will feel safe while using our application. So, the data that we will use, it will NOT be made public, it will be used only for filtrations, no more. The only Information that will be made public is the user's score, for our Scoreboard. But this score will be seen only if he is in Top 10. Otherwise, it will be seen only by him.

6.4 Appendix A: Glossary

  • HTML5 = Hyper Text Markup Language
    1. HTML is the standard markup language for creating Web pages
    2. HTML describes the structure of a Web page
    3. HTML consists of a series of elements
    4. HTML elements tell the browser how to display the content
    5. HTML5 is the fifth and final major HTML version that is a World Wide Web Consortium (W3C) recommendation
  • CSS = Cascading Style Sheets
    1. CSS describes how HTML elements are to be displayed on screen, paper, or in other media
    2. CSS saves a lot of work. It can control the layout of multiple web pages all at once
    3. External stylesheets are stored in CSS files
  • PHP = PHP: Hypertext Preprocessor
    1. PHP can generate dynamic page content
    2. PHP can create, open, read, write, delete, and close files on the server
    3. PHP can collect form data
    4. PHP can send and receive cookies
    5. PHP can add, delete, modify data in your database
    6. PHP can be used to control user-access
    7. PHP can encrypt data
  • SQL = Structured Query Language
    1. SQL lets you access and manipulate databases
